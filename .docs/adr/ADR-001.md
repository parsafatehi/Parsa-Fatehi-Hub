# ADR-001: Static-Site Generation with Client-Side Data Fetching for a Personal Web Hub

## 1. Context and Problem Statement

### 1.1 Technical Background

The system is a single-page, static personal homepage designed to serve as a central hub for an individual's online presence. The primary functional requirement is to present a Bento-style grid of cards, each linking to an external profile (e.g., GitHub, Spotify, Instagram). A secondary requirement is to display dynamic, "live" content within specific cards (e.g., a GitHub contribution graph, currently playing Spotify track, recently watched Trakt items). The system MUST be deployable as a static website on GitHub Pages without a backend server.

Historically, personal profile pages have been either purely static documents (e.g., Linktree-style pages) or dynamically rendered server-side applications. The former lacks live data integration, while the latter introduces operational overhead and complexity for a project with a single user and a low-traffic profile. The goal is to achieve a "best of both worlds" approach: a fast, reliable static site that can still incorporate real-time or near-real-time data from third-party services.

### 1.2 Current State Metrics

- **Build Time:** The current proof-of-concept HTML file builds in < 1 second.
- **Lighthouse Performance Score (Initial):** The current implementation scores > 95 for Performance, Accessibility, and Best Practices, primarily due to its static nature and minimal resource loading.
- **Data Update Latency:** The dynamic content (e.g., Spotify "Now Playing") is currently hardcoded. The target is for this data to be updated within a maximum of 5 minutes of a change occurring on the source platform (GitHub, Spotify, etc.). This is a key requirement for a "living" hub.
- **Deployment Complexity:** The current approach of a single HTML file has a deployment cycle time of under 30 seconds to push to GitHub Pages.

## 2. Decision Drivers

- **Driver A (Deployment Simplicity):** The system MUST be deployable to a standard static hosting platform (GitHub Pages) without the need for server-side infrastructure, serverless functions, or complex build pipelines beyond a standard static site generator.
- **Driver B (Performance & Reliability):** The homepage MUST load quickly (< 1.5 seconds on a 3G connection) and MUST have a high level of uptime and availability, as it is a personal hub. This precludes reliance on external APIs during the initial page load.
- **Driver C (Maintainability):** The codebase SHOULD require minimal maintenance. The chosen architecture SHOULD NOT introduce dependency on a specific cloud provider or require frequent updates due to API changes, thereby ensuring long-term reliability for the sole maintainer.
- **Driver D (Data Freshness):** The system SHOULD display up-to-date content from third-party services. Stale data (e.g., a week-old "Currently Playing" track) negatively impacts the "alive" feel of the application and reduces its utility.

## 3. Considered Options

### Option 1: Fully Static HTML with Periodic Rebuilds

- **Technical Overview:** This approach involves using a static site generator (SSG) to fetch data from all third-party APIs at build time and compile the entire page into a static HTML file. The build process is triggered on a schedule (e.g., via GitHub Actions) to refresh the data.
- **Pros:**
  * **Deployment Simplicity:** The final artifact is a static HTML file, which is perfectly compatible with GitHub Pages and has zero runtime dependencies.
  * **Performance & Reliability:** Page load times are exceptionally fast (optimal) as no client-side or server-side API calls are made. The page is fully loaded on the first request.
  * **Maintainability:** The architecture is simple and does not require server-side code or complex state management.
- **Cons:**
  * **Data Freshness:** The data is only as fresh as the last build. With a build schedule (e.g., every 15 minutes), there is a trade-off between freshness and resource usage. More frequent builds increase the computational load on the CI/CD pipeline.
  * **Scalability & Rate Limits:** While unlikely for a personal page, a high build frequency could approach API rate limits, and rebuilding the entire site for a single data point is inefficient.

### Option 2: Client-Side Data Fetching

- **Technical Overview:** This approach renders a static skeleton HTML file, and all dynamic content is fetched by JavaScript running in the user's browser after the page loads. The system would make direct API calls or fetch from a lightweight data proxy.
- **Pros:**
  * **Data Freshness:** This provides the best possible data freshness, as content is always fetched on page load.
  * **Deployment Simplicity:** The frontend remains a static asset that can be hosted on GitHub Pages.
- **Cons:**
  * **Performance & Reliability:** This violates the "performance" driver. It introduces multiple client-side API requests that can block rendering, increase page load time (e.g., "flash of empty content"), and fail due to CORS or network issues, making the page appear broken.
  * **Maintainability:** Exposing API keys on the client-side is a security risk and many services (e.g., Spotify, GitHub) do not allow direct browser-side calls without an OAuth flow or a proxy, increasing complexity.

### Option 3: Hybrid Approach (Static Shell + Serverless Function / Edge Data)

- **Technical Overview:** The core HTML and CSS are served statically, but the dynamic data is fetched client-side from a lightweight serverless function or an edge worker (e.g., Cloudflare Workers). This function acts as a secure proxy, handling API keys and fetching data from third-party services on behalf of the client.
- **Pros:**
  * **Data Freshness:** The data is fetched per user request, ensuring it is always up-to-date.
  * **Performance & Reliability:** The serverless function is geographically distributed (if using an edge network), reducing latency compared to a single origin server. The core page skeleton loads immediately, providing a good user experience (similar to Option 2, but with better reliability).
  * **Maintainability & Security:** API keys are stored securely on the serverless platform, not in the client-side code. The function can also aggregate data from multiple sources into a single API call, simplifying the client's logic.
- **Cons:**
  * **Deployment Simplicity:** This adds a significant external dependency on a serverless platform (e.g., Vercel, Cloudflare), which creates a more complex deployment pipeline and is a potential single point of failure.
  * **Cost & Complexity:** While typically low-cost for low-traffic personal projects, it introduces a potential cost and an additional technology (e.g., Workers, Vercel Functions) to manage and maintain, violating the "maintainability" and "deployment simplicity" drivers.

## 4. Decision Outcome

Chosen Option: **Option 1: Fully Static HTML with Periodic Rebuilds**, because it provides the best combination of performance, reliability, and simplicity for a personal project with a single maintainer. While it introduces a trade-off in "data freshness," this is deemed acceptable when weighed against the operational overhead and complexity introduced by Options 2 and 3. By using a generous build schedule (e.g., every 15 minutes), the system can achieve near-real-time data freshness while maintaining the ideal deployment and performance characteristics of a fully static site.

### 4.1 RFC 2119 Compliance Directive

All software engineering execution items resulting from this architecture decision MUST strictly comply with the [IETF RFC 2119](https://tools.ietf.org/html/rfc2119) standard phrasing. The tokens **MUST**, **MUST NOT**, **REQUIRED**, **SHOULD**, and **MAY** dictate absolute protocol criteria in the requirements below.

### 4.2 Core Engineering Requirements

1. The final build artifact **MUST** be a single directory containing only static assets (HTML, CSS, JavaScript, images) and **MUST** be deployable to GitHub Pages without any modifications.
2. The build process **MUST** be executable within a GitHub Actions workflow and **MUST** run on a predefined schedule (cron) to generate new static pages with updated data.
3. The build process **SHOULD** implement a caching layer (e.g., file-based caching) to store API responses from third-party services (GitHub, Spotify, Trakt) to reduce API calls and improve build speed.
4. The build process **MUST NOT** include any hardcoded secrets in the generated HTML files. API keys **MAY** be stored as GitHub Secrets and used exclusively during the build phase.
5. Client-side JavaScript **SHOULD** be minimized to enhance initial load performance, with all dynamic content pre-rendered into the static HTML.

## 5. Validation, Tracking, and Lifecycle

### 5.1 Verification Plan

- **Method:** The success of this architecture will be validated through a combination of manual inspection and automated CI checks.
- **Success Metric:** The system is considered successful if the following conditions are met:
    1.  **Deployment:** A new build is automatically triggered via GitHub Actions and deployed to GitHub Pages without manual intervention.
    2.  **Data Freshness:** A manual inspection of the deployed site after the scheduled build shows that data from external services (e.g., "recently played" on Spotify) is updated to reflect the current state.
    3.  **Performance:** The deployed site achieves a Lighthouse Performance score of > 95.
